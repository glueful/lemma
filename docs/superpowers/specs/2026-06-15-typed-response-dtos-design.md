# Typed Response DTOs — Retire the OpenAPI Finalize Script

**Goal:** Make the framework's reflect-only OpenAPI generator produce an accurate
enough Lemma spec on its own that `scripts/openapi-finalize-security.php` can be
deleted — by documenting the stable response shapes as doc-only `ResponseData`
DTOs and closing one small framework schema gap.

**Status:** Design approved (brainstorming). Next step: implementation plan.

---

## Background

The Lemma OpenAPI spec is generated by `composer docs:openapi`
(`php glueful generate:openapi -f --clean`). Historically a post-processor,
`scripts/openapi-finalize-security.php`, patched everything the reflect generator
could not express. Prior work in this session already made most of it redundant:

- Per-operation **security** is now native (delivery uses the `api_key`
  middleware → `ApiKeyAuth`; admin uses `auth` → `BearerAuth`; preview is public).
- **Error bodies** are native (`#[ApiResponse(4xx/5xx, schema: ErrorResponse::class,
  envelope: false)]` across all content controllers, plus a 500 on every action).
- **Request-body** schemas/array-items are native (`#[ArrayOf]` on request DTOs).
- **Path scoping** is native once the skeleton sample routes were deleted
  (`include_framework_routes=false`, `include_extensions=false`).

The script is already **unwired** from `docs:openapi` (kept only as the optional
`composer docs:finalize`). What remains script-only is **response-side polish**:
`date-time` formats, the `status` enum, nullable fields, response array `items`,
and required envelope keys. This effort reproduces those natively, then deletes
the script.

## Scope decisions (settled during brainstorming)

1. **Doc-only DTOs**, not real return-typed DTOs. Controllers keep returning their
   current arrays; we annotate `#[ApiResponse(200, SomeData::class)]`. This is the
   established `ErrorResponse` pattern — zero runtime/behaviour change. Lemma's
   payloads are content-model-driven, so even "real" DTOs would carry
   `array<string,mixed>` for the important part; the extra enforcement is not worth
   the refactor risk. Real return-typed DTOs are deferred for small **fixed**
   endpoints later (e.g. status, preview-token mint, delete confirmations).

2. **Type the stable shell, never the dynamic content.** Entry/delivery/preview
   payloads carry a per-content-type `fields` map that no DTO can statically
   describe. Type the fixed contract around it (`uuid`, `content_type`, `locale`,
   `status`, timestamps, cursor/list keys) and represent `fields` honestly as a
   freeform object.

3. **Two small framework generator fixes** (both generally-useful, not Lemma
   patches): (a) mark `success`/`message`/`data` `required` on the success
   envelope — they are always emitted, so the schema should say so; (b) make
   `ClassSchemaReflector` honor `#[ArrayOf]` in **response** mode (it currently
   reads it only in request mode), so `#[ArrayOf]` is consistently authoritative
   for array items in both request and response DTOs.

4. **Drop, consciously:** request-body examples, path-param descriptions, 429/ETag
   **headers**, and the nanoid `pattern` regex. None are required for a valid,
   agent-usable spec (confirmed: the framework has no `#[Header]`/`#[PathParam]`
   attribute and no field-level `pattern` hint; adding those is out of scope). The
   bare `304 Not Modified` response is kept cheaply via `#[ApiResponse(304)]`.

5. **Defer dangling-schema pruning** to a separate framework task. Deleting the
   script leaves 21 injected-but-unused component schemas (Error,
   ValidationErrorResponse, Webhook*, …). They are valid, just noise. (This noise
   already exists today, since `docs:openapi` no longer runs the script.) A blanket
   prune in the generator risks other apps that *do* reference those defaults, so
   it is scoped separately.

## Architecture

Doc-only `ResponseData` schema-holder classes describe the `data` object that the
framework's success envelope wraps. Where a controller wraps its payload in a named
key, the DTO models that key exactly. Example:

```php
// runtime: { "success": true, "message": "...", "data": { "content_type": { ... } } }
final class ContentTypeResultData implements ResponseData
{
    public function __construct(
        public readonly ContentTypeData $content_type,
    ) {}
}
```

How the reflector renders each field (verified against
`Glueful\Support\Documentation\ClassSchemaReflector`):

| DTO field type | OpenAPI output |
|---|---|
| `?DateTimeInterface` | `{type: string, format: date-time, nullable: true}` |
| backed enum (e.g. `EntryStatus`) | `{type: string, enum: [...]}` |
| `?string` / `?int` | scalar + `nullable: true` |
| `#[ArrayOf(FieldSchemaData::class)] array` | `{type: array, items: {…}}` — **requires framework fix (b)**; until it lands, `@var FieldSchemaData[]` is the response-mode source |
| `object` (the dynamic `fields`) | `{type: object}` — honest freeform object |
| nested `ResponseData` class | recursed object schema |

`object` is the deliberate choice for `fields`: the DTO is never constructed, so
the type hint need not match the runtime `array`; it yields a clean `{type: object}`
instead of the misleading `{type: array}` a bare `array` would produce.

## The DTO set

One class per file (PSR-4), grouped by domain folder under
`app/Content/Http/DTOs/Responses/`.

Model DTOs mirror the **exact raw row** the repository returns — controllers return
rows unchanged, so the drift-guard tests demand a key-for-key match. The columns
below are confirmed against the migrations + repository methods. The two items
still pinned during planning are `FieldSchemaData`'s exact field set and the
`PreviewData` / `PreviewMintData` payloads (from `PreviewReader` / `PreviewMinter`).

### `Responses/ContentTypes/`
Source: `001_CreateContentTypesTable` + `ContentTypeRepository::hydrate()` (decodes
`schema`, casts `schema_version`).
- **`ContentTypeData`** — `int id`, `string uuid`, `string slug`, `string name`,
  `?string description`, `#[ArrayOf(FieldSchemaData::class)] array schema`,
  `int schema_version`, `?string created_by`, `DateTimeInterface created_at`,
  `?DateTimeInterface updated_at`. **No `status`** — `content_types` has no such
  column.
- **`FieldSchemaData`** — `string name`, `string type` (backed enum:
  string/text/integer/number/boolean/datetime/reference/asset), `?bool required`,
  `?bool filterable`, … (mirrors the stored field shape; reuse the request-side
  `FieldDefinitionData` only if it reflects cleanly in response mode, else a
  dedicated class).
- **`ContentTypeResultData`** `{ContentTypeData content_type}` — store/show/updateSchema.
- **`ContentTypeListData`** `{ContentTypeData[] content_types}` — index.

### `Responses/Entries/`
Source: `002_CreateEntriesTable` / `003_CreateEntryDraftsTable` +
`EntryRepository::findEntry()`/`findDraft()` (both return the raw row; `findDraft`
JSON-decodes `fields`).
- **`EntryData`** — `int id`, `string uuid`, `string content_type_uuid`,
  `EntryStatus status`, `?string created_by`, `DateTimeInterface created_at`,
  `?DateTimeInterface updated_at`. **No `schema_version`** — that lives on the
  draft, not the entry.
- **`DraftData`** — `int id`, `string entry_uuid`, `string locale`, `object fields`,
  `int schema_version`, `int lock_version`, `?string updated_by`,
  `DateTimeInterface updated_at`. **No `created_at`** column on drafts.
- **`EntryCreateResultData`** `{EntryData entry, DraftData draft}` — store (201).
- **`EntryResultData`** `{EntryData entry}` — show.
- **`DraftResultData`** `{DraftData draft}` — getDraft / saveDraft.

### `Responses/Publication/`
- **`VersionResultData`** `{?string version_uuid}` — publish / rollback.
- unpublish returns empty `data` → left description-only (no DTO).

### `Responses/Preview/`
- **`PreviewMintData`** `{string token, ?DateTimeInterface expires_at, int expires_in}` — mint.
- **`PreviewData`** — `?string version_uuid`, `?int version`, `object fields`
  (exact payload from `PreviewReader::read()`, confirmed in planning).
- **`PreviewResultData`** `{PreviewData preview}` — read.

### `Responses/Delivery/`
- **`DeliveryItemData`** — `?string uuid`, `?string locale`, `?int version`,
  `?DateTimeInterface published_at`, `object fields`. Kept **separate** from
  `EntryData`: delivery serves a published-only public shape that will diverge
  from admin's draft/status view.
- **`DeliveryListData`** `{DeliveryItemData[] items, ?string next_cursor}` — index
  (cursor mode, the documented default). show returns a bare `DeliveryItemData`.

### Shared enums
- **`EntryStatus`** (backed string enum: `active`/`archived`/`deleted`), under
  `app/Content/Enums/` (or reuse an existing enum if present). `EntryData`
  references it so the reflector emits `enum: [...]`. Delivery's `item()` omits
  `status`, so `DeliveryItemData` carries none.

## Framework changes

Two additive fixes to the OpenAPI generator, both correct for every consumer:

**(a) Envelope `required`.** `RouteReflectionDocGenerator::wrapInEnvelope()` emits
`{type: object, properties: {success, message, data}}` with no `required`. Add
`'required' => ['success', 'message', 'data']` (mirroring the flat-pagination
envelope, which already marks them). Those keys are always present at runtime.

**(b) `#[ArrayOf]` in response mode.** `ClassSchemaReflector::arraySchema()` reads
`#[ArrayOf]` only when `$requestMode` is true; response mode falls back to the
`@var Foo[]` / `@var array<Foo>` docblock. Change response mode to **prefer
`#[ArrayOf]` when present** (resolving scalar/enum/DTO items, recursing in response
mode) and fall back to `@var` otherwise. Makes `#[ArrayOf]` consistently
authoritative for array items in both request and response DTOs.

Each fix includes a generator test and a `CHANGELOG [Unreleased]` entry.

### Sequencing (cross-repo)

1. **Framework:** add both fixes (envelope `required` + response-mode `#[ArrayOf]`)
   + tests + CHANGELOG.
2. **Framework release:** fold into a pending release if one exists; otherwise a
   real patch/minor, since Lemma is pinned to released `^1.58.0`. Do **not** invent
   a version number for unreleased work.
3. **Lemma:** bump the `glueful/framework` pin to the release containing the fix.
4. **Lemma:** add the response DTOs + `#[ApiResponse(200, X::class)]` annotations +
   drift-guard tests.
5. **Lemma:** delete `scripts/openapi-finalize-security.php` and the `docs:finalize`
   composer entry.

## Drift-guard tests

Doc-only DTOs are not compiler-enforced, so an integration test asserts each
documented endpoint's real `data` keys match its DTO. A reusable helper:

```php
// in LemmaTestCase or a trait
protected function assertDataMatchesDtoShape(array $data, string $dtoClass): void
{
    $params = array_map(
        fn (\ReflectionParameter $p) => $p->getName(),
        (new \ReflectionMethod($dtoClass, '__construct'))->getParameters()
    );
    $actual = array_keys($data);
    sort($params);
    sort($actual);
    self::assertSame($params, $actual);
}
```

Rules:
- Recurse **one level** (wrapper → model: assert `content_type`'s keys against
  `ContentTypeData`), but **never** into the freeform `fields` object.
- For `#[ArrayOf]` list wrappers (`ContentTypeListData.content_types`,
  `DeliveryListData.items`): assert each item — or at least the first non-empty
  item — against the item DTO. Again, never recurse into an item's `fields`.
- Extend the existing `tests/Integration/Http/*ApiTest.php`: each endpoint that
  carries a documented 200/201 asserts its real `data` against the DTO.

## Out of scope / follow-ups

- **Dangling-schema prune** — separate framework task (item 5 above).
- **nanoid `pattern`, request examples, path-param descriptions, 429/ETag headers** —
  dropped; would require new framework attributes (`#[Schema(pattern:)]`,
  `#[PathParam]`, `#[Header]`) if ever wanted.
- **Real return-typed DTOs** for small fixed endpoints (status/mint/delete) — a
  later, opt-in refinement.
- **Response shaping to drop internal columns** (`id`, `created_by`, `updated_by`
  on the admin payloads) — the DTOs document these because controllers currently
  return raw rows unchanged. Filtering them out is a deliberate API change for
  later, not part of this doc-only pass.

## Success criteria

- `composer docs:openapi` (no script) produces a spec where every **DTO-covered**
  2xx body (i.e. the stable shapes selected in this spec — excluding intentionally
  description-only responses like unpublish and the framework's special pagination
  mode) carries a typed `data` schema with native `date-time` formats, `status`
  enums, nullable fields, response array `items`, and required envelope keys.
- A diff of the no-script spec against the last script-finalized spec shows **only**
  the consciously-dropped items differ (examples, path-param descriptions,
  429/ETag headers, nanoid pattern, dangling schemas).
- Drift-guard tests pass; full Lemma suite + phpcs green.
- `scripts/openapi-finalize-security.php` and `docs:finalize` are removed.
